## 二、项目定位与核心设计

ds4.c 是一个**为 DeepSeek V4 Flash 模型深度定制的本地推理引擎**，不是通用 GGUF 加载器。

### 2.1 三大核心理念

1. **单一模型深度优化** — 只跑 DS4 Flash，不做通用框架
2. **Metal 独占** — macOS only，CPU 路径仅用于正确性验证
3. **KV Cache 磁盘化** — 压缩后的 KV cache 作为"一等磁盘公民"，利用 NVMe SSD 替代 RAM

### 2.2 Rust 端口架构

Rust 端口（rs/）用纯 Rust 重写了核心推理路径：

```
ds4 (binary)
├── GgufModel::open()           → GGUF 文件加载
├── bind_weights_unchecked()     → 权重绑定（零拷贝 mmap）
├── Vocab::load()               → Tokenizer 加载
├── build_chat_prompt()         → Chat 模板渲染（deepseek 格式）
├── Session::new()              → 创建推理会话
│   ├── session.sync()          → 预填充 KV cache
│   ├── session.argmax()        → Greedy 解码
│   └── session.sample()        → 采样解码
└── run_interactive()           → 交互式模式
```

依赖极简（仅 memmap2, bytemuck, anyhow），适合交叉编译到嵌入式平台。

## 三、三个核心问题回答

### (1) 能否在 Android 上运行？

**不能。** 三大壁垒：

| 障碍 | 说明 |
|------|------|
| Metal 依赖 | macOS-only GPU 加速。非 macOS 编译 `-DDS4_NO_METAL` 用 CPU 路径但性能极差 |
| 内存需求 | 2-bit 量化需要 **128GB RAM**（q2 模型 81GB + KV cache 20GB+） |
| 无 Android 构建 | Makefile 和 Cargo 都没有 Android target 配置 |

**结论**：DS4 Flash 本地推理与 Android 无缘。

### (2) 和 llama.cpp 比有什么优势？

| 维度 | llama.cpp | ds4.c |
|------|-----------|-------|
| 通用性 | 100+ 模型 | 仅 DS4 Flash |
| 量化策略 | 通用方案 | DS4 专属不对称量化（仅 MoE expert 量化） |
| KV Cache | 全部在 RAM | **磁盘持久化**（NVMe SSD） |
| MoE 优化 | 通用 MoE | 针对性 routed MoE（仅 up/gate IQ2_XXS, down Q2_K） |
| 验证方式 | 回归测试 | **与官方 logits 逐 token 比对** |
| 开发生态 | 60k+ stars | 7 stars (Alpha) |
| 社区成熟度 | 极高 | 实验性 |

**ds4 的真正价值**：不是替代 llama.cpp，而是证明：
1. "为单一模型深度定制"可以跑出更好的 2-bit 量化质量
2. KV Cache 磁盘化是新范式

### (3) API 接口是什么样？

双协议 HTTP Server（ds4-server）：

| 端点 | 协议 | 用途 |
|------|------|------|
| `GET /v1/models` | OpenAI | 模型列表 |
| `POST /v1/chat/completions` | OpenAI | Chat，支持 tools/tool_choice |
| `POST /v1/completions` | OpenAI | 文本补全 |
| `POST /v1/messages` | Anthropic | Claude Code 兼容 |

支持参数：temperature, top_p, top_k, min_p, seed, max_tokens, stream

## 四、对 Hermes_Rust_Operit_App 的可复用点

| 复用点 | 描述 |
|--------|------|
| **Rust 重写范式** | rs/ 的 Rust 端口证明了从 C→Rust 零拷贝 GGUF 加载可行性 |
| **KV cache 磁盘化** | 可借鉴到 Operit 的长上下文记忆管理 |
| **不对称量化策略** | 只量化 MoE expert、保留关键层精度的思路可移植 |
| **极简依赖** | ds4 Rust 端口仅 3 个依赖，适合嵌入式 |
| **API Server 架构** | 单线程串联 + 共享 checkpoint 的轻量设计 |

## 五、建议

- **不推荐**在 Operit 上直接使用 ds4（Metal-only + 128GB 起）
- 但其 Rust 端口（rs/）的 GGUF 加载和 tokenizer 代码可以抽取作为 Operit 的模型支持基础
- KV Cache 磁盘化的思路可以借鉴到 Operit 的记忆系统
